Web Bluetooth has been "the future" since 2017 when Chrome shipped it. Adoption is narrow because of fragmented support (Safari + Firefox refuse) and the security model is strict. But for the use cases it fits — hardware demos, BLE debugging, fitness device integration — nothing else comes close to a native browser API for talking to physical devices.

## Hello device

```js
const device = await navigator.bluetooth.requestDevice({
  filters: [{ services: ["heart_rate"] }]
});

const server = await device.gatt.connect();
const service = await server.getPrimaryService("heart_rate");
const char = await service.getCharacteristic("heart_rate_measurement");

await char.startNotifications();
char.addEventListener("characteristicvaluechanged", (e) => {
  const value = e.target.value.getUint8(1);
  console.log("BPM:", value);
});
```

Browser shows a native picker listing nearby BLE devices matching the filter. User picks one. Connection opens. Real-time notifications stream in.

## The security model

- **User-gesture required** — must be triggered by a click, not page load
- **Per-device permission** — granted only for THIS specific device, not "all heart-rate devices"
- **Origin-bound** — `https://example.com` can't access devices paired by `https://other.com`
- **HTTPS required** (no localhost exception in this case)
- **No background access** — connection drops when tab is hidden

These restrictions are why Web Bluetooth never became a general-purpose IoT API. They're also why it doesn't have a security history.

## When it shines

- **Conference demos / hardware booths** — visitors pair their wearable, watch their heart rate live in the browser
- **Internal IoT dashboards** — engineers debugging custom BLE peripherals without installing apps
- **Fitness apps** — pairing a heart-rate strap, cadence sensor, smart bike
- **Smart-home setup pages** — initial device pairing flows for routers, switches
- **Educational** — physics labs reading sensor data into a Jupyter-like web notebook

## When it fails

- **Cross-browser apps** — Safari + Firefox refuse to ship it; Chrome-only is the realistic ceiling
- **Mobile reach** — works on Chrome for Android, never on iOS Safari
- **Background workflows** — "sync when tab closed" doesn't work
- **Unknown devices** — without a service-name filter, picker is empty (no "show me all BLE devices" mode)
- **Anything Apple** — AirPods, Apple Watch, etc. don't speak standard BLE profiles

## Reading + writing characteristics

```js
// Read once
const value = await char.readValue();
console.log(value.getUint8(0));

// Write
const data = new Uint8Array([0x01, 0xff]);
await char.writeValue(data);

// Notify (push from device)
await char.startNotifications();
char.addEventListener("characteristicvaluechanged", e => {
  // event.target.value is a DataView
});
```

`DataView` for the value — same API as ArrayBuffer reads. `getUint8`, `getInt16`, `getFloat32`, etc. The BLE GATT spec defines the format per characteristic.

## GATT services + characteristics

Standard service names:
- `heart_rate`
- `battery_service`
- `cycling_speed_and_cadence`
- `running_speed_and_cadence`
- `health_thermometer`
- `device_information`

Custom services use 128-bit UUIDs. Vendor docs tell you which UUID maps to which characteristic.

## Disconnection handling

```js
device.addEventListener("gattserverdisconnected", () => {
  console.log("Device disconnected");
  // Optionally: try to reconnect
  setTimeout(async () => {
    try {
      await device.gatt.connect();
    } catch (e) { /* device out of range or off */ }
  }, 1000);
});
```

BLE connections drop frequently — battery, range, interference. Always handle disconnect; never assume the connection persists.

## When NOT to ship Web Bluetooth as your only path

If your app NEEDS device pairing for primary functionality:
- iOS users can't use it
- Firefox users can't use it
- ~30-40% of your audience excluded depending on demo

Use Web Bluetooth as **enhancement** ("for advanced users on Chrome, also available in our native app"). Don't make it required.

## Browser support

- Chrome / Edge 56+ (March 2017)
- Chrome for Android 56+ (March 2017)
- Opera 43+
- Safari: refused (Apple cites security concerns + iOS sandbox model)
- Firefox: refused (security stance)
- Samsung Internet 6+

iOS users CAN sometimes use it via Bluefy browser app, but that's a niche third-party.

## Tooling

- **chrome://bluetooth-internals** — debug page in Chrome for inspecting connections
- **nRF Connect** — companion mobile app for sending custom packets
- **Device Studio** — vendor BLE explorers like Nordic's

## What this enables

- Browser-side firmware updates for custom hardware
- Live demos at events / conferences
- Internal tools at IoT companies
- Educational hardware labs (Arduino BLE, micro:bit)

## What it doesn't

- Mass-market consumer apps (cross-browser problem)
- Background-sync workflows
- Multi-device orchestration without per-device user clicks

The pragmatic posture in 2026: Web Bluetooth is one of the platform's coolest niche features. Pick it for the use cases that fit. Don't bet a product on it without a fallback.
