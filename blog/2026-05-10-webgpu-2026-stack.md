WebGL was a port of OpenGL ES 2.0 — a 2007 mobile API frozen in JavaScript. It shipped wide, but compute shaders were missing, validation was painful, and modern GPU features (multi-draw, push constants, indirect dispatch) had no story. WebGPU is the replacement, modeled on Vulkan/Metal/D3D12. By 2026 it's in Chrome, Safari, and Firefox — and worth knowing about even if you don't build games.

## Why care if you're not Unreal Engine

WebGPU isn't just for 3D. The interesting workloads in browsers are:

- **ML inference** — running ONNX / Whisper / Stable Diffusion locally on the GPU
- **Image/video processing** — filters, resize, denoise at 60fps
- **Data visualization** — instanced point clouds with millions of vertices
- **Compute** — physics, particle systems, signed distance fields, parallel reductions
- **Cryptography** — hash, sign, verify in parallel

For these, WebGPU is **5–50x faster** than WebGL or WASM-SIMD on the same CPU.

## Hello compute shader

```js
const adapter = await navigator.gpu.requestAdapter();
const device = await adapter.requestDevice();

const shader = device.createShaderModule({
  code: `
    @group(0) @binding(0) var<storage, read> input: array<f32>;
    @group(0) @binding(1) var<storage, read_write> output: array<f32>;

    @compute @workgroup_size(64)
    fn main(@builtin(global_invocation_id) id: vec3<u32>) {
      let i = id.x;
      output[i] = input[i] * 2.0;
    }
  `
});
```

That's a GPU function: doubles every element of an array, 64 elements per workgroup, dispatched across thousands of workgroups in parallel.

```js
const pipeline = device.createComputePipeline({
  layout: "auto",
  compute: { module: shader, entryPoint: "main" }
});

const inBuf = device.createBuffer({
  size: data.byteLength,
  usage: GPUBufferUsage.STORAGE | GPUBufferUsage.COPY_DST
});
device.queue.writeBuffer(inBuf, 0, data);

const outBuf = device.createBuffer({
  size: data.byteLength,
  usage: GPUBufferUsage.STORAGE | GPUBufferUsage.COPY_SRC
});

const bindGroup = device.createBindGroup({
  layout: pipeline.getBindGroupLayout(0),
  entries: [
    { binding: 0, resource: { buffer: inBuf } },
    { binding: 1, resource: { buffer: outBuf } }
  ]
});

const encoder = device.createCommandEncoder();
const pass = encoder.beginComputePass();
pass.setPipeline(pipeline);
pass.setBindGroup(0, bindGroup);
pass.dispatchWorkgroups(Math.ceil(data.length / 64));
pass.end();
device.queue.submit([encoder.finish()]);
```

A million-element array doubled in microseconds. CPU sees the result via a readback staging buffer.

## WGSL — the shader language

WebGPU uses **WGSL** (WebGPU Shading Language), a Rust-flavored language designed for the spec. It's not GLSL, not HLSL — but it transpiles cleanly to/from both.

```wgsl
struct Vertex {
  @location(0) position: vec3<f32>,
  @location(1) color: vec3<f32>,
}

struct VertexOut {
  @builtin(position) clip_position: vec4<f32>,
  @location(0) color: vec3<f32>,
}

@vertex
fn vs_main(in: Vertex) -> VertexOut {
  var out: VertexOut;
  out.clip_position = vec4(in.position, 1.0);
  out.color = in.color;
  return out;
}

@fragment
fn fs_main(in: VertexOut) -> @location(0) vec4<f32> {
  return vec4(in.color, 1.0);
}
```

If you've written GLSL, this reads as expected. If you haven't, the structure is: input → struct → output → struct, with `@location(N)` declaring binding slots.

## Real wins over WebGL

| Capability                          | WebGL 2 | WebGPU |
|-------------------------------------|---------|--------|
| Compute shaders                     | ❌      | ✓      |
| Storage buffers (SSBO)              | ❌      | ✓      |
| Indirect draw                       | partial | ✓      |
| Multiple color attachments          | up to 4 | up to 8 |
| Texture arrays                      | partial | full   |
| Push constants                      | ❌      | ✓      |
| Bind groups (vs uniform-by-uniform) | ❌      | ✓      |
| Validation errors at queue time     | ❌      | ✓      |

The validation alone is huge — WebGL silently misrenders on bad shader bindings, WebGPU throws a JS exception with a precise error message at the queue.submit() call.

## Practical libraries

Most apps shouldn't write WGSL by hand. Pick a layer:

- **three.js** has a WebGPU renderer (`WebGPURenderer`) since r150 — same API as WebGL, with compute access via TSL (TypeScript Shading Language)
- **@webgpu/utils** for low-level buffer/bindgroup boilerplate
- **WebGPU API for ML**: `@webgpu/onnxruntime-web` runs ONNX models on the GPU
- **WebLLM**: runs Llama-class models entirely in the browser via WebGPU
- **Compute.toys** for shader experimentation (like ShaderToy but for compute)

## ML inference real numbers

Running Whisper-tiny (~40MB) for speech-to-text:

| Backend       | RTF (real-time factor) | Notes |
|---------------|------------------------|-------|
| CPU (WASM)    | 0.5x (slow)            | scalar |
| WASM-SIMD     | 1.2x (real-time)       | quantized i8 |
| **WebGPU**    | **6.5x (faster than real-time)** | f16, runs on integrated graphics |

Same model, ~13x speedup over scalar WASM. WebGPU is what makes browser-side AI feel responsive.

## When NOT to use WebGPU

- **Static UI rendering** — CSS/canvas2D is faster to develop and uses less memory
- **Tiny one-shot computations** — buffer setup costs ~5ms, eats wins on small jobs
- **Apps targeting only old iPhones** — WebGPU on iOS Safari shipped in 18, older devices don't get it
- **Code shipped to first-time visitors with strict bundle limits** — three.js + your shaders adds a couple hundred KB

## The cost: GPU memory

WebGPU buffers live in dedicated GPU memory. On a 4GB integrated chip you can easily exhaust it with sloppy allocation. Always destroy buffers when done:

```js
const buf = device.createBuffer({ ... });
// ... use buf ...
buf.destroy();  // free GPU memory immediately
```

Without `destroy()`, GC eventually reclaims, but on memory-pressured devices that means OOM crashes.

## Browser support

- Chrome / Edge 113+ (April 2023)
- Safari 18+ (September 2024)
- Firefox 121+ (December 2023, behind flag until 2024)

For non-supporting browsers, fall back to WebGL or WASM. The capability check:

```js
if (!navigator.gpu) {
  console.log("WebGPU unavailable — fallback");
}
```

## What this enables

Local Whisper transcription. Browser-side Stable Diffusion. Real-time video filters. Particle simulations with millions of particles. ML demos that don't ship data to a server. The Electron-app pile that justified itself with "needs GPU" shrinks every year.

For most apps it's an over-investment today. For the ones it fits, it's the difference between "wait for the cloud" and "do it now in the tab."
